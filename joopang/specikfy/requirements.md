# Git 리포지토리 기반 API 설계 관리
API 설계 문서를 Git에서 관리하면 버전 추적, 협업, 리뷰가 가능합니다. 이 문서들은 단순한 개발용이 아니라 비즈니스 로직 설계와 클라이언트 소통에도 활용됩니다.
추천 디렉토리 구조

# 각 문서의 역할
requirements.md: 비즈니스 요구사항 정의
스테이크홀더 요구사항
기능적/비기능적 요구사항
비즈니스 규칙 및 제약사항

user-stories.md: 사용자 관점의 기능 명세
클라이언트와의 소통 도구로 활용
인수테스트 시나리오 기반

api-specification.md: 기술적 API 명세
엔드포인트와 HTTP 메소드
요청/응답 데이터 구조
개발팀 내부 레퍼런스로 활용

data-models.md: 비즈니스 로직 설계
엔티티 관계 정의
비즈니스 규칙 반영
DB 설계와 비즈니스 로직의 거울

# 개발 워크플로우와 생성 문서:
1. 요구사항 분석 → requirements.md
2. 사용자 스토리 작성 → user-stories.md
3. API 설계 → api-specification.md
4. 문서 생성 → openapi.yaml
5. 코드 구현 → 실제 코드
6. 테스트 작성 → 테스트 코드
7. 배포 → 프로덕션 환경
8. 각 단계별로 문서를 생성하여 체계적으로 개발을 진행합니다.

## 1단계: 요구사항 분석
사용자 스토리 → 기능 명세 → API 엔드포인트

## 2단계: API 설계
- 자원 식별 및 URL 구조 설계
- HTTP 메소드 매핑
- 요청/응답 데이터 구조 정의

## 3단계: 코드 생성
- API 명세서 작성
- 검증 로직 구현
- 에러 처리 구현

## 효과적인 프롬프트 예시
1단계: 요구사항 문서 생성
```
할일 관리 앱의 요구사항을 분석해서 docs/api/requirements.md 파일을 작성해주세요.
기능적 요구사항과 비기능적 요구사항을 구분하고,
사용자 역할과 권한도 포함해주세요.
```

### 2단계: 사용자 스토리 생성
```
requirements.md를 기반으로 user-stories.md를 작성해주세요.
As a [role], I want [feature], so that [benefit] 형식으로 작성하고,
각 스토리마다 인수 조건(Acceptance Criteria)도 포함해주세요.
```
​
### 3단계: API 명세서 생성
```
user-stories.md를 기반으로 RESTful API 명세서를 작성해주세요. api-specification.md 에요
엔드포인트, HTTP 메소드, 요청/응답 형식을 포함하고,
에러 처리와 상태 코드도 정의해주세요.
```
​
### 4단계: 데이터 모델 설계
```
API 명세서를 기반으로 데이터 모델을 설계해주세요. data-models.md에요.
Entity Relationship Diagram과 각 엔티티의 속성,
제약조건, 인덱스 정보를 포함해주세요.
```
​
### 5단계: OpenAPI 스펙 생성
API 명세서를 OpenAPI 3.0 형식으로 변환해주세요. docs/api 같은 폴더에요
Swagger UI에서 테스트할 수 있도록 예시 데이터와
인증 방식도 포함해주세요.
