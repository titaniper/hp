
# 왜 시스템 설계가 중요한가요?
API 개발은 단순히 코드를 작성하는 것이 아닙니다. 전체 시스템의 구조를 이해하고, 데이터의 흐름을 파악하며, 각 컴포넌트 간의 상호작용을 설계하는 것이 핵심입니다.

## 좋은 시스템 설계는:
- 유지보수성: 나중에 수정하기 쉬운 구조
- 확장성: 사용자가 늘어나도 안정적으로 동작
- 가독성: 팀원들이 쉽게 이해할 수 있는 구조
- 협업 효율성: 프론트엔드, 백엔드, DevOps 팀 간 원활한 소통

# 시스템 설계의 3가지 핵심 도구
1. 인프라 구조도: "어떤 서버와 서비스들이 어떻게 연결되어 있는가?"
2. ERD (Entity Relationship Diagram): "데이터는 어떻게 구조화되고 관계를 맺는가?"
3. 시퀀스 다이어그램: "요청이 들어왔을 때 시스템 내부에서 어떤 순서로 처리되는가?"

## 설계 단계
1. 요구사항 분석 → ERD 초안 작성
2. ERD 리뷰 → 팀원들과 데이터 구조 검토
3. API 설계 → 시퀀스 다이어그램으로 동작 흐름 정의
4. 인프라 설계 → 시스템 전체 구조 확정

## 개발 단계
1. ERD를 기반으로 데이터베이스 스키마 생성
2. 시퀀스 다이어그램을 참고하여 API 로직 구현
3. 인프라 구조에 따라 배포 환경 설정

## 유지보수 단계
1. 새 기능 추가 시 설계 문서 먼저 업데이트
2. 변경사항이 다른 컴포넌트에 미치는 영향 분석
3. 성능 문제 발생 시 시퀀스 다이어그램으로 병목 지점 파악

이러한 시스템 설계 기초를 탄탄히 다져놓으면, 더 복잡한 서비스도 체계적으로 개발할 수 있습니다!

🧭 제품 문서의 단계별 구조
단계	문서 이름	주요 작성자	초점	산출물 예시
1️⃣	요구사항 정의서 (Requirements Definition)	기획자 / PM / 고객	“무엇을 해결해야 하는가”	고객 요구, 문제 정의, 목표, 제약사항
2️⃣	PRD (Product Requirements Document)	PM / PO	“무엇을 만들 것인가”	핵심 기능, 사용자 시나리오, 성공 기준
3️⃣	기능 명세서 (Functional Specification)	기획자 / 개발 리드	“어떻게 동작할 것인가”	기능 ID, 입력/출력, 플로우, 예외처리
4️⃣	기술 설계서 (Technical Design Doc, TDD)	개발자 / 아키텍트	“어떻게 구현할 것인가”	DB 스키마, API 구조, 컴포넌트 다이어그램
5️⃣	테스트 케이스 / QA 시나리오	QA / Tester	“제대로 동작하는가”	시나리오별 입력값, 기대 결과, Pass/Fail 기준




## 🧭 요약 비교표

| 구분     | **BRD** (Business Requirements Document)                    | **PRD** (Product Requirements Document)                            |
| ------ | ----------------------------------------------------------- | ------------------------------------------------------------------ |
| 초점     | **왜** 이걸 해야 하는가                                             | **무엇**을 만들 것인가                                                     |
| 관점     | **비즈니스 / 전략 중심**                                            | **제품 / 사용자 경험 중심**                                                 |
| 주요 작성자 | 경영진, 기획자, PO, 사업 담당자                                        | PM, PO, 제품기획자                                                      |
| 대상 독자  | 경영진, 이해관계자(Stakeholder)                                     | 개발팀, 디자이너, QA, 마케팅팀                                                |
| 포함 내용  | - 시장 배경<br>- 문제 정의<br>- 비즈니스 목표<br>- KPI / ROI<br>- 프로젝트 범위 | - 주요 기능<br>- 사용자 시나리오<br>- 성공 기준 (Metric)<br>- UX 플로우<br>- 기능 우선순위 |
| 결과물    | “이걸 하면 어떤 가치가 생길까?”                                         | “이걸 어떤 형태의 제품으로 구현할까?”                                             |
| 예시 문장  | “고객 이탈률을 20% 줄이기 위해 자동 리마인드 시스템을 도입한다.”                     | “사용자가 결제 후 3일 이내 미입금이면 자동 푸시 알림을 발송한다.”                            |

## 📘 예시로 보는 차이 (커피 원두 판매 서비스)

| 구분         | 내용                                                                                                            |
| ---------- | ------------------------------------------------------------------------------------------------------------- |
| **BRD 예시** | - **문제:** 고객이 재구매 시 어떤 원두를 샀는지 기억하지 못함.<br>- **비즈니스 목표:** 재구매율 15% 증가.<br>- **해결 방향:** 고객 구매 이력 기반 추천 기능 추가.  |
| **PRD 예시** | - **기능명:** 개인화 원두 추천<br>- **시나리오:** 사용자가 로그인 → 이전 구매 기록 기반으로 3개 원두 추천<br>- **성공 지표:** 추천 클릭률 30%, 재구매율 15% 상승 |

👉 즉, **BRD는 ‘왜 만들지’**, **PRD는 ‘무엇을 만들지’**를 정의합니다.

---

## 🧩 실제 흐름 관계

```
BRD (왜 필요한가)
     ↓
PRD (무엇을 만들 것인가)
     ↓
FS / FSD (어떻게 동작해야 하는가)
     ↓
TDD / API Spec (어떻게 구현할 것인가)
```

## 💡 비유로 이해하기

| 단계  | 비유                                                 |
| --- | -------------------------------------------------- |
| BRD | “우리가 커피숍을 차려야 하는 이유는 사람들이 빠르게 테이크아웃 커피를 원하기 때문이다.” |
| PRD | “그렇다면 키오스크 주문, 빠른 결제, 주문 현황 표시 기능이 필요하다.”          |

---

## 🚀 요약 한 줄 정리

> **BRD**는 “비즈니스 논리와 목표를 정의하는 문서”
> **PRD**는 “그 목표를 실현할 제품의 요구사항을 정의하는 문서”
