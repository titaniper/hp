# 0. Discovery (문제 발굴 · 가설 설정)

**목적:** 무엇을 왜 하는지 정리, 성공 기준 합의

* **산출물**

  1. **BRD (Business Requirements Document / 사업 요구사항 정의서)**

     * 시장/문제 진술, 대상 고객, 비즈니스 목표(KPI/ROI), 성공/실패 조건, 리스크, 범위/비범위
  2. **Stakeholder Map & RACI** (의사결정 구조, 책임 배분)
  3. **가설 캔버스** (Problem → Hypothesis → Metrics → Risks)

* **책임:** PO/PM, 사업담당, 데이터/리서처(옵션)

* **게이트(통과 기준):** 비즈니스 목표와 성공지표가 수치로 정의됨(예: 재구매율 +15%), 범위/비범위 합의

**팁:** “우리는 어떤 행동 변화를 일으켜야 KPI가 오른다고 믿는가?”를 한 줄로 적어두세요.

---

# 1. Definition (제품 정의)

**목적:** 무엇을 만들지, 누구에게 어떤 가치로 줄지 결정

* **산출물**

  1. **PRD (Product Requirements Document / 제품 요구사항 문서)**

     * 타깃 페르소나/세그먼트
     * 핵심 문제·가치 제안(Why)
     * **핵심 기능(Features) & 우선순위** (MoSCoW/ICE/RICE)
     * 상위 **유저 시나리오/저니** (AS-IS/TO-BE)
     * 성공 지표(제품 KPI), 롤아웃 전략(MVP → GA)
  2. **Use Cases & User Stories**

     * EPIC → Capability → User Story(“As a … I want … so that …”)
     * **Acceptance Criteria**(Gherkin/BDD 권장)
  3. **Non-Functional Requirements(NFR)**

     * 성능, 가용성, 보안/개인정보, 접근성, 로깅/모니터링, 국제화, 규제

* **책임:** PM/PO 주도, 기획/디자인/개발 리드 합의

* **게이트:** 각 기능의 **수용 기준**이 명문화되고, MVP 범위·우선순위가 동의됨

**팁:** PRD에는 “안 하는 것(Out of scope)”을 반드시 적어 스코프 크리프를 막으세요.

---

# 2. Solution Design (해결안 설계)

**목적:** 기능을 시스템 동작과 설계로 구체화

* **산출물**

  1. **FSD/FS (Functional Spec / 기능 명세서)**

     * **기능 ID, 트리 구조, 흐름(상호작용), 예외 케이스, 에지 조건**
     * 화면 이벤트 → 백엔드 호출 → 응답/오류 패턴까지 연결
  2. **IA & UX 산출물**

     * 정보구조(IA), 네비게이션 맵, **와이어프레임/플로우 다이어그램**, 프로토타입(Lo/Mid-Fi)
  3. **시스템 아키텍처 설계서(High-level TDD)**

     * 컨텍스트 다이어그램, 컴포넌트/서비스 경계(DDD 바운디드 컨텍스트),
       동기/비동기, 데이터 흐름(이벤트/메시지), 확장성·캐시 전략
  4. **데이터/API 명세**

     * **ERD & 스키마 버전 전략**, 마이그레이션 계획
     * **API 스펙(OpenAPI/Swagger)**: 경로, 파라미터, 응답, 에러 코드 규약
     * 권한/인증(JWT/OAuth), 레이트리밋, 감사로그
  5. **시퀀스/상태 다이어그램**

     * 주문→결제→재고→정산 같은 **긴 트랜잭션 경계**와 롤백 시나리오, 락/일관성 모델
  6. **테크 리뷰 메모**

     * 기술 리스크, 의존성, 선택과 배제의 근거(Design Rationale)

* **책임:** 아키텍트/Tech Lead, BE/FE 리드, QA, 보안/개인정보 담당

* **게이트:** 핵심 플로우에 대한 **시퀀스/에러 처리**와 **API/ERD**가 동결(버전 1.0)되고, NFR 대응 설계가 있음

**팁:** 시퀀스 다이어그램에 **트랜잭션 시작/커밋/보상(보상 트랜잭션)** 표시를 남겨두면 구현·운영 이슈가 크게 줄어요.

---

# 3. Delivery Planning (전달 계획 · 일정/비용)

**목적:** 만들 순서, 리소스, 일정과 리스크를 수치로 확보

* **산출물**

  1. **WBS & 백로그** (EPIC→Story→Task)

     * 각 항목 **스토리포인트/MD** 추정, 의존성 표기
  2. **릴리즈 로드맵 & 마일스톤**

     * MVP → Beta → GA, **기능 토글/점진 롤아웃** 계획
  3. **테스트 전략 & 품질 기준**

     * 테스트 피라미드(단위/통합/계약/ E2E), 커버리지, 성능/부하 기준
  4. **보안·개인정보 영향평가 체크리스트**, **접근성 점검 기준**

* **책임:** PM/스크럼마스터, Tech Lead, QA Lead

* **게이트:** 일정·범위·리소스 트레이드오프 합의, **Definition of Ready(DoR)** 정의

**팁:** 계약 테스트(Consumer-Driven Contract)로 FE/BE 병렬 개발 속도를 올리세요.

---

# 4. Implementation (개발)

**목적:** 설계대로 만들고 지속 통합/검증

* **산출물**

  1. **개발 가이드 & 코딩 규칙** (Lint/Format, 브랜치 전략, 커밋 규약, 모듈 경계)
  2. **API 문서 자동화**(Swagger UI), **스키마 마이그레이션 스크립트**
  3. **테스트 코드**

     * 단위/통합/계약/E2E, **시나리오→AC(수용기준)** 매핑
  4. **CI/CD 파이프라인**

     * 빌드·테스트·보안 스캔·배포, 환경별 설정(Dev/Stage/Prod)
  5. **관측성(Observability) 구성**

     * 로그 스키마, 트레이싱(분산 트레이스), 대시보드, 알람 임계값

* **책임:** 개발팀 전원, DevOps/SRE, QA

* **게이트:** **Definition of Done(DoD)** 충족(테스트 통과, 커버리지, 보안 스캔, 문서화)

**팁:** 스키마와 API는 **버저닝**하고, 실서버에는 **기능 토글**로 위험을 낮춥니다.

---

# 5. Verification (검증 · 릴리즈 준비)

**목적:** 요구·기능·설계가 의도대로 동작하는지 확인

* **산출물**

  1. **테스트 케이스/시나리오 & 결과 리포트**(AC 기준 Pass/Fail)
  2. **성능/부하/장애 복원 테스트**(SLA/SLO 검증)
  3. **보안·개인정보 점검 결과**(취약점 조치)
  4. **릴리즈 노트 & 배포 체크리스트**
  5. **롤백/핫픽스 계획**

* **책임:** QA Lead, SRE/DevOps, PM/Tech Lead 승인

* **게이트:** KPI/품질 기준 충족, **릴리즈 승인(Go/No-Go)**

**팁:** 실제 운영 데이터 분포를 반영한 **샘플링·마스킹 데이터**로 성능 테스트를 돌리세요.

---

# 6. Launch & Operation (출시 · 운영)

**목적:** 안정 배포·모니터링·학습 루프

* **산출물**

  1. **런북/플레이북**(장애 유형별 절차, 알람 응답 체인, 대체 플로우)
  2. **온콜/알림 라우팅 정책**, **대시보드**(KPI, 에러율, 지연)
  3. **분석 이벤트 설계 & 대시보드**(Ampli/Mixpanel/GA)
  4. **CS FAQ & 내부 운영 매뉴얼**(권한·정산·환불 등)
  5. **포스트모템 & 리드아웃**(학습 항목 → 백로그 피드백)

* **책임:** 운영/CS, SRE, 데이터, PM

* **게이트:** 초기 KPI 추적, 안정 기간 SLO 준수

**팁:** 출시 직후는 **기능 토글로 점진적 확대**(1%→10%→100%), CS·에러 로그와 같은 날 모니터링 룸 운영.

## 한눈에 보는 문서 순서 (체크리스트)

1. **BRD** → 목표·문제·KPI 정의
2. **PRD** → 페르소나/가치/기능·우선순위/지표
3. **Use Cases & User Stories + AC**
4. **NFR** (성능·보안·가용성·로그·접근성)
5. **FSD/FS** (기능 명세: ID/흐름/예외)
6. **IA/와이어/프로토타입**
7. **아키텍처 설계서** (컴포넌트/데이터·이벤트/경계)
8. **ERD & API 스펙(OpenAPI)** + 시퀀스/상태 다이어그램
9. **WBS/백로그/견적 & 로드맵** + 테스트 전략
10. **개발 가이드/CI/CD/관측성 설계**
11. **테스트 케이스 & 결과** + 성능/보안 리포트
12. **릴리즈 노트/배포 체크리스트/롤백 계획**
13. **런북/운영 매뉴얼/분석 대시보드**
14. **포스트모템 & 개선 백로그**

## 샘플 섹션 템플릿(요약)

* **PRD-Feature 카드**

  * ID/이름/가치(Why)/설명(What)/AC/우선순위/Risk/Metric/Owner
* **FSD-기능 표**

  * 기능ID | 시나리오 단계 | 입력/출력 | 상태변화 | 예외 | 호출 API | 권한
* **API 오류 규약**

  * `code`/`message`/`hint`/`recoverable`/추적ID
